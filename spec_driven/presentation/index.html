Le Problème

Pourquoi le vibe coding ne suffit plus

🎲

Vibe Coding

Instructions vagues, "prompt and pray" — l'IA produit du code "à peu près correct" qu'il faut constamment corriger et re-prompter

🏚️

Dérive silencieuse

L'IA fait des choix implicites (librairies, architecture, conventions). Après 3 mois, le projet est un "legacy IA"

❓

Perte de savoir

Sans documentation de l'intention, les features deviennent des mystères. Le code existe mais personne ne sait POURQUOI

→ Le SDD fait de la spec — pas du code — la source de vérité. Le code est bon marché. La clarté est chère.
Concept

Qu'est-ce que le Spec-Driven Development ?

Spec existante → Nouveau besoin → Mise à jour spec → Implémentation IA

Le SDD est une discipline d'ingénierie où l'on rédige des spécifications détaillées AVANT d'écrire du code. Ce n'est PAS "une spec par tâche" — c'est "l'app entière est spécifiée". Les specs sont la documentation vivante permanente du système.

🎲 Vibe Coding
Décrire au fur et à mesure — rapide mais fragile
Le code EST l'artefact principal
📐 Spec-Driven
Spécifier d'abord — plus lent au départ, fiable en production
La spec EST l'artefact principal
💡 Le SDD est le parent en amont de TDD. TDD = "tests d'abord". SDD = "intention d'abord — puis dérive tests ET code".

Layering

La pyramide des specs

🏛️ Niveau Projet Rarement modifié
CLAUDE.md / Rules ADRs schema.prisma openapi.yaml
📦 Niveau Module / Domaine Évolue avec le domaine
coaching-plans/SPEC.md user-management/SPEC.md billing/SPEC.md
🔧 Niveau Feature / Changement À chaque itération
.specs/feat--email-auth/ Requirements + Design + Tasks
L'IA lit de haut en bas ↓
⚠️ Sans le niveau Projet, l'IA réinvente l'architecture. Sans le niveau Module, elle ignore les règles métier. Sans le niveau Feature, elle improvise.

Découverte

Connecter les specs à l'IA

Avec des dizaines de specs, tout charger = gaspillage de tokens. Solution : une découverte en 3 tiers.

1 CLAUDE.md Toujours chargé

Pointeur léger + index des specs (~5-10 lignes)

## Specs
- DB: prisma/schema.prisma
- API: docs/api/openapi.yaml
- ADRs: docs/adrs/
- Modules have SPEC.md files

Before implementing, read the
relevant SPEC.md first.
Coût contexte : quasi nul
2 Rule agent-decided Chargée si pertinent

Workflow SDD complet dans une Rule

# .cursor/rules/spec-workflow.mdc
# .claude/rules/spec-workflow.md
description: "When implementing
features or modifying archi"

→ Contient le workflow
  Explorer → Spécifier →
  Implémenter → Valider
Chargée uniquement quand pertinent
3 Skill /spec À l'invocation

Loop complet encapsulé dans une skill

# .cursor/skills/spec/SKILL.md
# .claude/skills/spec/SKILL.md

1. Explorer le code + specs
2. Rédiger/mettre à jour spec
3. Implémenter par tâche
4. Valider + boucler
Chargée uniquement quand invoquée
💡 Le CLAUDE.md est la table des matières, pas le livre entier. L'IA sait OÙ chercher sans tout charger.
Workflow

Le workflow SDD en 4 phases

1
Explorer
L'IA lit le code ET les specs existantes
Ne PAS écrire de code
→
2
Spécifier
Mettre à jour les specs ou créer une spec dans .specs/
Requirements Design Tasks
→
3
Implémenter
Déléguer chaque tâche à un sous-agent
Commit après chaque tâche
→
4
Boucler
Review, tests, et mettre à jour les specs permanentes
SPEC.md + schema + OpenAPI
→ La boucle se ferme quand les specs permanentes sont à jour. Sinon, la prochaine fois, l'IA travaillera avec des specs obsolètes.
Exemple Concret

Ajouter l'authentification par email

État initial Le module user-management existe mais sans authentification
user-management/SPEC.md Avant 
# User Management

## Modèle
- User: id, email, name
- CRUD basique via API REST

## Endpoints
GET  /api/users
POST /api/users
GET  /api/users/:id

⚠️ Pas d'auth, pas de sessions
prisma/schema.prisma Avant 
model User {
  id    String @id @default(uuid())
  email String @unique
  name  String
}

// Pas de modèle Session
// Pas de champ password
Spec de changement .specs/feat--email-auth/ — les 3 couches
.specs/feat--email-auth/requirements.md Requirements 
# Auth par Email — Requirements

## Problème
L'app n'a pas d'authentification.

## Acceptance Criteria (EARS)
QUAND un utilisateur soumet le formulaire
  AVEC un email valide,
  LE SYSTÈME DOIT créer un compte
  et retourner un token JWT.

SI l'utilisateur échoue 5 tentatives,
  LE SYSTÈME DOIT verrouiller le
  compte pendant 15 minutes.

## Hors scope
- Social login (Phase 2)
- 2FA (Phase 3)

EARS

Écrire des requirements avec EARS

Easy Approach to Requirements Syntax — popularisé par AWS Kiro. Des patterns simples pour des specs sans ambiguïté.

Ubiquitaire
LE SYSTÈME DOIT [capacité]
LE SYSTÈME DOIT hasher les mots de passe avec bcrypt
Événementiel
QUAND [événement] LE SYSTÈME DOIT [réponse]
QUAND un utilisateur soumet le formulaire d'inscription, LE SYSTÈME DOIT créer un compte et retourner un JWT
État
TANT QUE [état] LE SYSTÈME DOIT [comportement]
TANT QUE le compte est verrouillé, LE SYSTÈME DOIT rejeter toute tentative de connexion
Conditionnel
SI [condition] LE SYSTÈME DOIT [action]
SI l'utilisateur échoue 5 tentatives, LE SYSTÈME DOIT verrouiller le compte 15 minutes
Complexe
QUAND [evt] TANT QUE [état] SI [cond] LE SYSTÈME DOIT [réponse]
QUAND un login est tenté TANT QUE le rate-limit est actif SI l'IP est blacklistée, LE SYSTÈME DOIT retourner 429
Implémentation

Déléguer aux sous-agents

📐 Specs à jour
→
🧠 Agent principal lit les specs
→
⚡ Sous-agent A Task 1 → commit
⚡ Sous-agent B Task 2 → commit
⚡ Sous-agent C Task 3 → commit
🧹
Contexte propre
Pas de dégradation — chaque sous-agent repart à zéro
⚡
Parallélisation
Plusieurs tâches en même temps
↩️
Commits atomiques
Réversibles, review facile
🎯
Scope isolé
Chaque sous-agent reçoit les specs du module concerné
💡 1 tâche = 1 sous-agent = 1 commit — c'est la règle d'or de l'implémentation SDD.

Bonnes Pratiques

Les 8 règles d'or

1

Spec = source de vérité

L'app entière est spécifiée. Le code est dérivé des specs, pas l'inverse.

2

Maintenir la pyramide

Projet + Module + Feature. L'IA a besoin de tous les niveaux.

3

Organiser par domaine

Un dossier par feature métier. Meilleur scope pour l'IA.

4

Permanentes vs WIP

SPEC.md par module + .specs/ pour les changements en cours.

5

Boucler la boucle

Après implémentation, mettre à jour les specs permanentes. Sinon = IA perdue.

6

Contexte en 3 tiers

CLAUDE.md (pointeur) + Rule (workflow) + Skill (loop). Ne pas tout charger.

7

Tâches atomiques

Max 3 fichiers par tâche, definition of done, commit par tâche.

8

La spec est le prompt

Plus la spec est précise, meilleur est le code. Garbage in, garbage out.

📐 Récapitulatif

Le code est bon marché. La clarté est chère. Le SDD investit dans la clarté.
Les specs documentent TOUT le système : API, DB, ADRs, PRDs, modules, features
La pyramide : Projet > Module > Feature — l'IA lit de haut en bas
Workflow : Explorer → Spécifier → Implémenter → Boucler (mettre à jour les specs)
Combiné avec Rules, Commands/Skills et Hooks = workflow complet
Commencez par créer un SPEC.md pour votre module principal.
Puis spécifiez votre prochaine feature !
