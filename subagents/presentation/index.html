Le problème

Pourquoi déléguer ?

📣

Contexte saturé

Exploration du dépôt, logs shell, DOM navigateur : des sorties bruyantes qui encombrent le fil principal si tout reste dans une seule conversation.

⏳

Tout en série

Sans parallélisme, chaque étape attend la précédente. Sur un chantier large, le temps perdu augmente vite.

🎯

Rôles mélangés

Recherche, design, implémentation et vérif dans le même prompt → moins de clarté, plus d’erreurs et d’oublis de contraintes.

→ Les sous-agents isolent le bruit, permettent le parallèle et appliquent des spécialistes ciblés.
Les trois leviers

Parallélisme · Isolation · Spécialisation

🔲
Isolation
Contexte propre
Chaque sous-agent dispose de sa fenêtre de contexte. En général, le parent reçoit surtout une synthèse ou un résumé — les intermédiaires volumineux restent à côté (selon l’outil et le mode).
Doc Cursor & Anthropic
⚡
Parallélisme
Débit
Plusieurs délégations indépendantes en parallèle (ex. openapi.yaml + section Installation du README.md) au lieu d’attendre la fin de chaque étape.
Task / messages
🧩
Spécialisation
Qualité
Prompt système, outils et modèle adaptés au rôle : exploration en lecture seule, audit, tests, etc.
Même équipe, rôles différents
💡 Les docs officielles décrivent le même principe : une tâche annexe qui inonderait la conversation principale est un bon candidat à la délégation.
Mécanisme

Comment ça marche ?

Parent → Délégation + contexte utile → Sous-agent → Résumé / synthèse au parent

Le sous-agent repart d’un contexte propre : il n’a pas l’historique implicite du chat parent. Le parent doit donc fournir l’essentiel dans le message de délégation. Après exécution, c’est souvent une synthèse ou un résumé structuré qui remonte — le détail dépend de l’outil et du mode.

⏱️ Premier plan
Bloque jusqu’à la fin du sous-agent — idéal quand la suite du travail dépend du résultat.
Cursor : mode foreground (doc)
🌙 Arrière-plan
Le parent continue ; suivi ou reprise plus tard — idéal pour tâches longues ou branches parallèles.
Cursor : background + resume

Pas à pas (à l’écran)
1 Ouvrir le projet de démo ; montrer .cursor/agents/ et un fichier court (ex. verifier.md).
2 Montrer le frontmatter : name, description (intentions concrètes), readonly si agent de vérif.
3 Dans le chat Agent : /verifier ou une tâche qui déclenche la délégation auto selon la description.

Démo — Claude Code

Patterns

Trois patterns utiles

Cas d’usage Réduire les faux « terminé » : un sous-agent sceptique relit et teste.

Sous-agent en readonly (Cursor) ou outils lecture seule (Claude Code) : vérifier que l’implémentation existe, lancer les tests pertinents, signaler ce qui manque. À enchaîner après les tâches de la spec (lien avec la capsule Spec-driven).

Cas d’usage Grand chantier : découper en phases avec remises structurées.

Plan (recherche) → implémentation (general-purpose ou agent parent) → vérif. Chaque phase peut être déléguée pour garder le fil principal lisible.

Cas d’usage Deux travaux indépendants : même message, deux délégations.

Exemple concret : « Mettez à jour openapi.yaml pour refléter le nouveau endpoint et, en parallèle, réécrivez la section Installation du README.md. » Deux livrables indépendants, nommés — formulation qui favorise plusieurs sous-tâches lorsque l’outil le permet (voir doc Cursor sur le parallélisme).

Décision

Sous-agents ou Skills ?

🤖 Sous-agent
Isolation de contexte, parallélisme, expertise sur plusieurs étapes, ou besoin d’une vérification indépendante.
Recherche lourde, audits, branches longues
📎 Skill / commande
Action courte, répétable, un seul contexte suffit — ex. changelog, formatage ciblé (grille officielle Cursor).
Éviter la sur-délégation
⚠️ Anti-patterns (Cursor) : dizaines d’agents vagues, descriptions génériques (« aide au code »), prompts énormes. Mieux vaut 2–3 sous-agents clairs qu’une armée floue.
Bonnes pratiques

Six règles d’or

1

Un rôle par agent

Pas de « helper général » — une responsabilité, clairement décrite dans la description (signal de routage), sans alourdir le corps du prompt.

2

description = routage

C’est le signal principal pour la délégation automatique — investissez-y deux minutes.

3

Prompts courts

Corps du fichier : direct, testable ; évitez les pavés de 2 000 mots.

4

Versionner

.cursor/agents/ et .claude/agents/ dans Git pour l’équipe.

5

Coût conscient

Parallèle = plusieurs contextes ; inutile pour des micro-tâches.

6

Commencer intégré

Explore / Bash / Browser (Cursor) ; Explore / Plan (Claude Code) avant d’empiler du custom.

🎯 Récapitulatif

Déléguer sert surtout à protéger le contexte principal, paralléliser le travail indépendant et spécialiser les comportements.
Cursor et Claude Code partagent l’idée ; les chemins (.cursor/agents/, .claude/agents/, /agents, Task) se recoupent en partie.
Avec Rules, Skills, Hooks et Spec-driven : les sous-agents sont la couche d’exécution qui évite l’asphyxie du fil unique.
Créez un agent « vérif » en lecture seule et passez-le après votre prochaine feature.
