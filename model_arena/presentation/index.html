Décision

Tâche → orientation (API)

Noms d’exemples côté fournisseur — choisissez le SKU exact dans la doc du moment.

Scénario Orientation Pourquoi (règle de base)
Petit changement localisé, peu de contexte Modèle économique (ex. gpt-4o-mini, Claude Haiku 4.5) Minimiser $ et latence si le risque d’erreur est faible
Refactor large, architecture, beaucoup de fichiers Modèle fort (ex. Sonnet 4.6, gpt-4o) + grosse fenêtre Meilleure cohérence quand le contexte et la complexité explosent
Raisonnement difficile, debug bloqué Modèle raisonnement (ex. o3 / o3-mini) ou mode étendu documenté côté Claude Plus de compute / thinking tokens — inadapté au « micro-fragment » interactif
Très grosse base de code ou doc Fenêtre longue + résumé en amont, caching si répétitif Éviter de saturer le contexte : voir limites & pricing cache sur les docs
Pièges

Trois confusions à éviter

🧾

API vs abonnement

Le prix du forfait Claude / ChatGPT Plus ne s’applique pas à l’appel messages / chat.completions côté API

📤

Output > input

Un généré long (JSON, rapport) génère beaucoup de tokens de sortie — souvent plus chers

📊

Benchmark ≠ prod

Le « meilleur » sur un classement public n’est pas forcément le meilleur pour votre budget et votre latence

Conclusion

Contrainte d’abord, modèle ensuite

→ Budget, délai, risque d’erreur : fixez-les, puis choisissez le SKU. Mettez à jour les prix au fil des changements (trimestriel recommandé).
