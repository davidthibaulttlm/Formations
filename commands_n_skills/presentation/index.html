Le Problème

Pourquoi s'en servir en équipe ?

🔄

Équipe non standardisée

Chaque dev pilote l'IA à sa façon, les résultats divergent et les conventions se brisent.

😵

Rituel répétitif

Tannant de réécrire : "roule le linter, corrige pour que le CI passe". Une commande /lint le fait en un geste.

🐘

Trop de rules, trop de bruit

Le contexte gonfle, ça ralentit et coûte plus cher. Les skills chargent à la demande et regroupent docs/scripts/assets dans un seul dossier.

→ Rules + Commands + Skills versionnées dans Git = une façon officielle et partagée d'utiliser l'IA dans l'équipe

Boucle Agentique

Comment l'IA décide d'appeler une skill

1) Ce qui est dans le contexte

Pour la découverte, l'IA voit surtout le frontmatter du SKILL.md (name, description, contraintes).

2) Décision d'invocation

Dans sa boucle agentique, elle compare la demande utilisateur avec ces métadonnées pour décider d'installer/invoquer la skill.

3) Chargement ciblé

Si pertinent, la skill est chargée et exécutée ; sinon, elle n'est pas ajoutée au contexte pour éviter le bruit.

⚠️ Implication pratique : la qualité du frontmatter, surtout la description, influence directement la probabilité d'activation automatique.

# Lint Fix CI

Lance le linter du projet.
Corrige les erreurs détectées.
Relance le linter jusqu'à zéro erreur.
Résume les changements.

Si le lint passe, confirme que le CI
devrait passer sur ce point.
▶ Claude Code
1 Créez .claude/commands/ ou .claude/skills/
2 Ajoutez un frontmatter YAML
3 Utilisez $ARGUMENTS pour les paramètres
4 Utilisez !`cmd` pour injecter du contexte
.claude/commands/lint.md 
---
description: Lance lint et corrige les erreurs CI
argument-hint: [scope optionnel]
allowed-tools: Read, Edit, Bash
model: sonnet
---

Lance le linter sur $ARGUMENTS.
Contexte : !`git diff --name-only`
Corrige jusqu'à ce que ça passe.

Bonnes Pratiques

Les 6 règles d'or

1

Partir d'un irritant réel

Ex : "lint CI", "review PR", "deploy". Le meilleur point de départ est une friction répétée.

2

Pour du nouveau, privilégiez les skills

Créez une command seulement pour un raccourci manuel simple ; sinon, standardisez directement en skill.

3

Soigner la description

La description du frontmatter influence directement la décision d'invocation de l'IA.

4

SKILL.md < 500 lignes

Déplacer la doc détaillée dans references/ pour garder un contexte efficace.

5

Versionner dans Git

Rules/commands/skills dans le repo = standard équipe, reproductible et officiel.

6

Dossier de skill autonome

Un dossier par skill avec scripts/assets/references pour mieux maintenir et faire évoluer.
