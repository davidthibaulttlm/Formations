Définition

Model Context Protocol

Applications LLM + JSON-RPC 2.0 + Données & outils standardisés

Protocole ouvert pour brancher des assistants sur des sources de données, des APIs et des outils. Messages UTF-8, sessions avec état, négociation des capabilities. La spécification officielle vit sur modelcontextprotocol.io.

💡 Analogie pédagogique : le LSP a unifié les éditeurs autour des langages ; le MCP vise la même clarté pour le contexte et les actions des agents.
Architecture

Hôte, client, serveur

🖥️

Hôte (host)

L’application (IDE, client chat…) qui orchestre les sessions, applique les politiques de sécurité et le consentement utilisateur.

🔌

Client MCP

Composant dans l’hôte qui ouvre une connexion vers un serveur (souvent une connexion par serveur).

⚙️

Serveur MCP

Expose resources, tools, prompts — local (stdio) ou distant (HTTP streamable).

⚠️ La confiance se gère côté hôte : le protocole ne remplace pas une revue de sécurité ni une politique d’accès.

Implémentation

Créer un serveur en 3 étapes

SDK TypeScript officiel
1 McpServer — nom, version, instructions optionnelles
2 Enregistrer tools / resources / prompts (Zod v4 / Standard Schema)
3 StdioServerTransport (local) ou transport HTTP (distant) puis connect
// Paquets : @modelcontextprotocol/server, zod
import { McpServer, StdioServerTransport }
  from '@modelcontextprotocol/server';
import * as z from 'zod/v4';
Scaffolding
1 CLI create-typescript-server (dépôt MCP officiel)
2 Adapter les tools à votre domaine métier
3 Publié sur GitHub : modelcontextprotocol/create-typescript-server
# Exemple (voir doc du repo pour la commande exacte)
npx @modelcontextprotocol/create-typescript-server mon-serveur
💡 Les instructions serveur guident les workflows (ex. ordre des appels) — ne pas répéter ce qui est déjà dans chaque description d’outil.
Exemple

Un outil minimal + stdio

Pattern registerTool + handler async + StdioServerTransport
src/index.ts MCP TS SDK 
const server = new McpServer({ name: 'demo', version: '1.0.0' });

server.registerTool(
  'calculate-bmi',
  {
    title: 'IMC',
    description: 'Calcule l’indice de masse corporelle',
    inputSchema: z.object({
      weightKg: z.number(),
      heightM: z.number()
    }),
    outputSchema: z.object({ bmi: z.number() })
  },
  async ({ weightKg, heightM }) => {
    const output = { bmi: weightKg / (heightM * heightM) };
    return {
      content: [{ type: 'text', text: JSON.stringify(output) }],
      structuredContent: output
    };
  }
);

const transport = new StdioServerTransport();
await server.connect(transport);
💡 Schémas stricts = moins d’appels invalides ; structuredContent facilite les réponses typées côté client.
Équivalent SDK Python mcp — même trio : serveur, outil décoré, transport stdio
server.py MCP Python 
from mcp.server.fastmcp import FastMCP

mcp = FastMCP('demo')

@mcp.tool()
def calculate_bmi(weight_kg: float, height_m: float) -> dict:
    """Calcule l’IMC"""
    bmi = weight_kg / (height_m ** 2)
    return {'bmi': bmi}

if __name__ == '__main__':
    mcp.run()
💡 Les noms de packages et API évoluent — toujours vérifier le README officiel du SDK Python sur GitHub modelcontextprotocol/python-sdk.

Bonnes pratiques

7 règles d’or

1

Descriptions explicites

Le modèle choisit les outils d’après le texte — soignez titre et description.

2

Schémas stricts

Valider toutes les entrées ; erreurs claires si validation échoue.

3

Effets de bord

Documenter idempotence, risques, et ce qu’une tool modifie réellement.

4

Timeouts & erreurs

Réponses exploitables par l’agent ; éviter les stack traces brutes.

5

Moindre privilège

Exposer le minimum de ressources et d’outils nécessaires.

6

stdout propre (stdio)

Logger sur stderr ou fichier — jamais de bruit sur stdout.

7

Version & doc d’équipe

Nom, version, changelog — surtout en monorepo partagé.
